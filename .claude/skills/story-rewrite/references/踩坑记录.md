# story-rewrite 踩坑记录

> 历史问题与解法，供排查时参考。

### Windows 中文编码问题（2025-05-30）

**现象**：`python -c` 执行包含中文的三引号字符串时，PowerShell 编码与 Python 不一致，导致 `SyntaxError: unterminated triple-quoted string literal`。

**根因**：Windows PowerShell 5.1 默认编码（GBK/cp936）与 Python 的 UTF-8 不匹配，中文字符被截断。

**解法**：
- 文件写入 → 用 `write` 工具，不用 `bash` + `python -c`
- 字数统计 → 先 write 文件，再单独 `python wordcount.py` 读文件统计
- 绝不在 `python -c` 命令里包含多行中文字符串

### 多agent并行导致故事断裂（2025-05-30）

**现象**：多个agent并行写多章，每个agent自行设计关键事件，导致章节间时间线断裂、事件版本冲突、人物行为矛盾。

**根因**：Phase 2 prompt 没有锁定关键事件锚点，agent各自发挥。

**解法**：
- Phase 1 框架新增「关键事件锚点表」，主线程锁定每章必须发生的1-2个关键事件
- Phase 2 新增章纲生成步骤，将锚点展开为具体情节点
- Phase 3 prompt 注入章纲内容，agent在章纲内自由发挥

### story-rewrite 章纲格式优化（2025-05-30）

**问题**：story-rewrite 章纲只有2个关键事件+收尾钩子，粒度太粗。

**优化方案**：章纲格式包含：核心事件、情节点序列、目标情绪、章首钩子、爽点、章尾钩子（含期待度）、字数目标。

### opencode 逐字输出导致长文本慢（2025-05-30）

**解法**：agent 用 write 工具写入文件，不返回正文到终端。只返回状态信息。

### 章纲生成后文件未保存（2026-05-31）

**现象**：story-architect agent 返回了章纲内容，但没有写入文件。主线程手动保存，流程断裂。

**根因**：skill 只说"存储到文件"，没明确是谁存。agent 默认只返回内容。prompt里给的是相对路径，agent不知道项目根目录。

**解法**：
1. prompt 中用绝对路径：`C:\Users\Administrator\...\{书名}\大纲\章纲_全书.md`
2. 明确写"⚠️ 你必须用 write 工具写入文件"
3. 主线程 agent 返回后 Read 验证文件是否存在

### 字数修复依赖人工判断（2026-05-31）

**现象**：agent写的章节超标或不足，主线程只是报告给用户，没有自动修复。用户需要手动指示"截断""补写"。

**根因**：skill说"必须截断"但没给自动化流程，主线程不知道怎么截断。

**解法**：
1. 写后自动修复流程：主线程遍历文件，>3000用PowerShell按段落边界截断，<1400用task_id要求补写
2. 修复是自动的，看到不达标就直接修，不报告不询问
3. 修完后再汇报结果

### 多任务并行导致临时文件覆盖（2025-05-30）

**解法**：临时文件用新书名作前缀：`temp_{书名}_ch{N}.txt`

### 批量模式上下文溢出（2025-05-30）

**解法**：摘要压缩机制（近6章详记+更早概要+卷级总览），每批只注入压缩摘要。

### 朱雀AI检测AIGC值0.94-0.99（2025-05-30）

**现象**：rewrite产出的3章全文被朱雀AI检测标记为AI特征，AIGC值0.94-0.99。

**根因v1**：禁用词替换只能处理表层AI味，AIGC检测器看统计特征。

**解法v1（已证明无效）**：Phase 3 prompt 新增7条抗AIGC规则。问题：规则列表式约束对LLM无效。

**根因v2**：
1. 场景展开法三层结构本身是AI指纹
2. 段落结构过于均匀
3. 信息密度均匀
4. "禁止"式规则无法改变写作习惯

**解法v2（结构重写）**：
- 从"禁止AI模式"转向"强制注入人类写作特征"
- 新增写作前的结构模板（段落骨架规划、句长分布模板、句首轮换模板）
- 新增8条硬规则（单句段≥5个、信息跳跃≥3处、禁止机械三层展开等）
- 新增写后量化验证清单（10项检查，不达标则重写）
- 场景展开法从"必须三层"降级为"参考"

### 试水阶段字数约束失效（2025-05-30）

**现象**：3章正文全部超标（Ch1=3589, Ch2=3251, Ch3=5975）。

**解法v1**：字数约束提到 prompt 最高优先级，内置字数自检方法，Phase 4 主线程二次校验。

**解法v2（2026-05-31 补充）**：主线程拿到agent返回后必须立即截断，不能只标记"超标"就放过。Phase 2 强制流程新增"字数检查+截断"步骤，>3000字必须按段落边界截断。

**解法v3（2026-05-31 拆分后）**：试水逻辑拆到独立 skill `story-rewrite-preview`，主 skill `story-rewrite` 直接写全书，无试水阶段。字数校验统一用 PowerShell 批量处理。

### 朱雀AI检测 AIGC 0.93-0.99：结构完整度问题（2026-05-30）

**现象**：仿写《穿成炮灰替身后》3章全文朱雀检测，正文 AIGC 值 0.9318，框架部分 0.9979。

**根因分析**：
- 禁用词控制良好（几乎无命中），说明禁用词规则有效
- 核心问题是**选词概率分布太规律**——LLM 默认输出"高概率词"（工整、流畅、完整），朱雀识别的就是这种分布
- 具体被标记模式：内心独白完整推理链、段落长度均匀、对话标签规律交替、场景弧线完整
- 加更多规则列表无法改变分布，反而稀释注意力

**解法**（2026-05-31 更新）：
- Phase 3 prompt 删除所有规则列表
- 替换为：**语感样本 > 规则**。给 agent 源文本片段让它模仿，比告诉它"你是XX型人格"更有效
- 原理：模仿具体文本从源头改变选词分布，贴标签只是让 agent "表演"风格
- 关键教训：**人格注入会让 agent 表演风格（有规律=AIGC），语感模仿让 agent 自然流露（无规律=更像人）**

### 子agent执行卡顿：5个效率bug（2025-05-30）

**Bug 1：对"不是A"规则过度防御**
- agent 把"不是闹钟"误判为禁用的"不是A，而是B"句式
- 修复：禁用句式示例明确标注 ✅ 允许简单否定句

**Bug 2：手动逐字计数**
- agent 在思考中逐字数章节，多次重算，消耗大量 token
- 修复：新增"禁止手动逐字计数"规则，强制使用 PowerShell

**Bug 3：对话段落计数导致反复重构**
- agent 为满足"连续同句数段 ≤ 2段"规则，反复合并/拆分对话段落
- 修复：新增对话段落豁免规则

**Bug 4：角色名/地名在 prompt 中被淹没**
- prompt 框架太长，关键信息混在中间，agent 写错后才发现
- 修复：prompt 顶部新增「角色锚点」块

**Bug 5：发现问题后全文重写而非增量修改**
- agent 每发现一个违规词就全文重写，写了 5 个版本
- 修复：新增「增量修改」策略

### 多agent并行导致设定矛盾（2026-05-31）

**现象**：3个agent并行写3章，出现4处致命逻辑硬伤：
1. 门牌号矛盾（Ch1说403/404，Ch2说402）
2. 母亲离去方式矛盾（Ch1说留信封消失，Ch2说临终握着手交代）
3. 奶奶角色突兀降落（Ch1睡楼道，Ch3突然有亲奶奶住同栋楼）
4. 报警情节断层（Ch2报警收尾，Ch3完全不提后续）

**根因**：prompt没有共享硬设定，agent各自编造细节。并行写作无法感知其他agent的内容。

**解法**：
1. Phase 1 框架新增「设定锁」：锁死地名、门牌号、人物关系、背景事件、时间线
2. prompt 模板新增「设定锁」注入：所有agent共享同一份硬设定
3. prompt 模板新增「角色锚点」块：放在 prompt 顶部，防止角色名/地名被淹没
4. prompt 模板新增「跨章连续性」section：注入上章cliffhanger+未关闭事件
5. Phase 3 每 3 批跑一次 story-review 做一致性审查
6. **2026-05-31 更新**：每 agent 写 K 章（K≤3），同一 agent 内后续章天然知道前章内容，减少跨 agent 设定冲突

### AI套路句式渗透（2026-05-31）

**现象**：编辑指出25%疑似AI句子，具体模式：
- "脊背绷得笔直，像一根随时会折断的铁丝"
- "嘴角翘了一下。不是笑，是一种无奈的自嘲"
- "手指在动，脑子里却乱成一团"

**根因**：禁用词列表只管单个词汇，不管句式套路。这些句子没有命中任何禁用词，但结构高度模板化。

**解法**：prompt 新增「反AI套路句式」section，列出6种常见套路句式+替代方案+判断标准。

### 并行 agent 创造冲突设定（2026-05-31）

**现象**：第2章 agent 创造了"顾砚深，衡石律师事务所·合伙人"的设定，但第3章 agent 只拿到第1章摘要，自行编造了"名片上印的是技术总监，还是个什么创业孵化器的"。两章名片内容严重矛盾。

**根因**：第2-3章并行写作，第3章 agent 不知道第2章创造了什么具体设定。设定锁只锁了"有名片"这个动作，没锁名片上的具体内容。

**解法**：
1. Phase 2 写完后、合并前，**强制启动 consistency-checker subagent** 做一致性审查
2. 审查范围：人名/地名/职业/身份/道具内容/时间线/人物关系
3. 发现冲突 → 主线程直接改文本，不重跑 agent
4. 新增 CP-6 检查点：一致性审查未执行，禁止合并输出
5. 设定锁建议补充：关键道具的具体内容（如名片上写什么、信里说什么）也应纳入设定锁

### 设定锁粒度优化（2026-06-01）

**问题**：设定锁粒度不够细，只锁定了大类，没有锁定具体内容，导致多agent并行时出现设定冲突。

**优化方案**：
1. **增加"动态细节"分类**：在设定锁中新增"动态细节"部分，用于锁定写作过程中新创造的具体内容
2. **增加"道具内容"字段**：对于关键道具，锁定其具体内容（如名片上写什么、信里说什么）
3. **增加"锁定级别"**：
   - 一级锁定：绝对不可更改（如人物关系、核心事件）
   - 二级锁定：可以扩展但不能矛盾（如地名可以增加但不能改地址）
   - 三级锁定：可以自由发挥（如临时道具、对话内容）
4. **统一格式**：统一使用表格格式，便于agent解析，增加"来源"字段记录设定来源
5. **自动更新机制**：每批写作完成后，从正文中提取新创造的具体内容，补充到"动态细节"部分

**设定锁模板**：
```markdown
## 设定锁（所有agent共享，不可违反）

### 1. 核心地名（一级锁定）
| 地名 | 说明 | 锁定级别 |
|------|------|----------|
| {地名} | {具体地址和描述} | 一级 |

### 2. 关键物件（一级锁定）
| 物件 | 来源+状态 | 具体内容 | 锁定级别 |
|------|----------|----------|----------|
| {物件名} | {来源和状态} | {具体内容} | 一级 |

### 3. 人物关系锁（一级锁定）
| 角色 | 身份 | 关系 | 锁定级别 |
|------|------|------|----------|
| {角色名} | {身份} | {与其他角色的关系} | 一级 |

### 4. 背景事件锁（一级锁定）
| 事件 | 时间 | 描述 | 锁定级别 |
|------|------|------|----------|
| {事件名} | {时间} | {具体描述} | 一级 |

### 5. 时间线锁（一级锁定）
| 章节范围 | 时间 | 关键事件 | 锁定级别 |
|---------|------|---------|----------|
| {章节范围} | {时间} | {关键事件} | 一级 |

### 6. 动态细节（二级锁定）
| 类型 | 内容 | 来源章节 | 锁定级别 |
|------|------|----------|----------|
| 道具内容/{具体道具名} | {具体内容} | 第{N}章 | 二级 |
| 对话内容/{具体对话} | {具体内容} | 第{N}章 | 二级 |
| 场景细节/{具体场景} | {具体内容} | 第{N}章 | 二级 |
```

**更新规则**：
1. Phase 1 创建初始设定锁
2. 每批写作完成后，从正文中提取新创造的具体内容，补充到"动态细节"部分
3. 每批写作前，agent必须读取设定锁并核对关键信息
4. 发现冲突时，以设定锁为准，不得自行修改设定锁
