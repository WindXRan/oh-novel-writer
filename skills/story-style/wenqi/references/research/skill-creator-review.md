# Skill-Creator 评审：闻栖写作风格 SKILL.md

评审日期：2026-05-31
评审视角：skill-creator（Skill 可运行性与可靠性）

---

## 各维度评审结果

### 1. 激活触发条件 — ⚠️ 过宽，存在大量误激活风险

**问题**：trigger 中混合了「闻栖专属词」与「通用网文套路词」，后者会导致 Skill 在用户只想写普通甜宠文时被误激活。

| 没问题 | 有问题（太通用，易误激活） |
|--------|--------------------------|
| 闻栖风格 | 追妻火葬场 |
| 仿闻栖 | 番茄女频 |
| 闻栖式开篇 | 先婚后爱 |
| 闻栖式感情线 | 协议结婚 |
| 写闻栖味的小说 | 穿书甜宠文 |
| | 开局让女主被渣 |
| | 男主暗恋女主多年 |
| | 虐渣甜宠 |

**风险场景**：用户说「我想写个先婚后爱的甜宠文」→ Skill 激活，AI 强行套用闻栖的「一睁眼」开篇模板，但用户想要的可能是晋江慢热型先婚后爱，完全不匹配。

**缺少的触发词**：`--style=wenqi`（story-rewrite 的插件调用方式）、`wenqi`（无中文语境时兜底）、`番茄短章`、`闻栖节奏`。

### 2. 角色扮演规则的可操作性 — ❌ 严重缺失

SKILL.md 提供了 5 个心智模型 + 9 条启发式 + 4 套模板 + 质检清单，但没有**任何一行指引 AI 激活后应该按什么流程响应**。

**具体缺失**：
- 没有问题路由：用户说「帮我写个开篇」vs「帮我看看这章」vs「分析闻栖风格」，三种意图需要三种不同的响应流程，但 SKILL 没有区分
- 没有「收到用户请求后的第一步」: 是否需要先确认书名/主角/设定？是否需要先读用户已有的项目文件？
- 没有「首个回复」模板：AI 激活后第一句话应该是什么？
- 没有「用户输入不足时的澄清协议」：需要哪些最少信息才能开始写？
- 所有模板和模型都是静态参考材料，不是可执行指令

### 3. 失败预防 — ⚠️ 被动声明，无主动处理

「诚实边界」列出了 8 条限制性声明（不适用男频、不适用长章平台、不模拟真人等），但**全是信息性说明，不是可执行指令**。

**缺失**：
- 当用户要求写男频时，AI 应该怎么做？说「对不起不能写」然后终止？还是说「可以帮你调整成女频」？
- 当用户提供了 SKILL 覆盖不了的信息需求时（如「帮我画个人设图」），AI 应该怎么回应？
- 没有「优雅降级」路径

### 4. 频率约束 — ❌ 完全缺失

**无任何使用频率限制**。

- 没有「每章写完后自动暂停，请用户确认后再继续」
- 没有「全本写作需分阶段确认」的 checkpoint 机制
- 没有「单次最大生成字数」限制
- 没有「先输出模板/框架，用户确认后再展开」的防浪费机制

对比同类优质 Skill（如 story-rewrite 的「先询问再执行」模式），这个 Skill 在生成节制方面几乎为零。

### 5. 缺失的关键信息 — 用户仿写时还缺什么？

| 缺失信息 | 影响 |
|---------|------|
| **如何将用户的原创设定映射到闻栖框架**（「用户的男主是霸总」→「对应闻栖的暗恋型/隐忍型男主」的转换规则） | 用户不知道自己的角色/世界观怎么"翻译"成闻栖风格 |
| **至少 1 个完整的章节示范**（模板只给骨架，不给血肉） | 用户不知道「信息倒灌」在实际文本中长什么样，写出来比例失调 |
| **闻栖的目标读者画像与平台策略**（写给谁看？在哪发？什么数据算好？） | 用户不知道自己的仿写作在市场中的位置 |
| **story-rewrite 的集成说明**（这个 Skill 被 story-rewrite --style=wenqi 调用，但 SKILL 内零提及） | 外部调用路径断裂，用户无法通过正常命令使用 |
| **用户必须提供的最少信息清单** | AI 不知道要问什么，用户不知道要提供什么，对话效率低 |

## 具体改动建议（2-3 处）

### 改动 1：在「你是谁」下方插入显式的工作流协议

**问题**：全篇零执行流程。AI 激活后有知识无步骤。

**改后文本示例**（插入在 `你是谁` 小节之后，`核心写作心智模型` 之前）：

```markdown
## 激活工作流

你被激活后，严格执行以下三步流程：

### Step 1：识别用户意图（路由）

| 用户说… | 路由到… | 要求 |
|---------|---------|------|
| 「帮我写/仿写/生成…」 | 写作模式 | 必须先确认书名/主角名/核心设定，再用模板A/B/C/D |
| 「帮我改/修/润色/去AI味…」 | 修改模式 | 必须先读用户提供的文本，再对照质检清单逐项检查 |
| 「分析/拆解/拆书/看看…」 | 分析模式 | 输出结构分析 + 风格匹配度评分，不下场改写 |
| 「闻栖的特点/风格分析…」 | 教学模式 | 用表达DNA和心智模型回答问题，不生成小说文本 |
| 模糊请求（「帮我写个故事」） | 先澄清 | 第一步是问清书名/题材/主角等最少信息 |

### Step 2：最少信息收集（仅在写作模式下）

在生成任何文本前，必须确认以下最少信息：
- [ ] 书名/作品名
- [ ] 女主核心设定（姓名+性格关键词）
- [ ] 男主核心设定（姓名+性格关键词）
- [ ] 穿书/重生/先婚后爱？是哪种子类型？
- [ ] 是否有用户已有的设定文件/大纲？→ 优先读取

信息不足时，主动输出澄清问题，不猜测。

### Step 3：Checkpoint 节流

- 每次最多生成 1 章 → 等待用户确认/修改
- 不允许一次性输出「前 30 章」或全文
- 用户要求批量生成时，输出章节大纲框架让用户确认后再逐章执行
- 违背上述规则 = Skill 执行错误
```

---

### 改动 2：收紧触发词 + 增加 fallback 兜底

**问题**：trigger 混入过多通用套路词，误激活率高；无 `--style=wenqi` 触发路径。

**改后文本示例**（修改 frontmatter 中的 trigger 字段）：

```yaml
trigger: >
  当用户通过 `--style=wenqi` 参数调用 story-rewrite、或明确提到「闻栖风格」「仿闻栖」「闻栖式开篇」「闻栖式感情线」「写闻栖味的小说」时激活。
  也适用于用户说「用闻栖的节奏写」「闻栖式女频」「闻栖的对话方式」「像闻栖那样写追妻火葬场/契约婚姻」时激活。
  退化触发（仅在其他女频Skill未激活时）：用户说「番茄女频短章写法」「女频开篇绑定模板」「短章发糖节奏」但未指定风格时，可作为兜底。
  不要在用户只说「写个甜宠文」「先婚后爱」「穿书文」等通用女频需求时激活——先确认用户是否明确要闻栖风格，再做路由。
```

---

### 改动 3：在「诚实边界」之后插入使用约束与优雅降级

**问题**：无频率约束，无越界处理，用户说「分析闻栖的真实身份」时 SKILL 无应对方案。

**改后文本示例**（追加在 `诚实边界` 小节末尾）：

```markdown
### 使用约束

1. **单次最大输出**：1 章（2,000-3,000 字）。如需批量生成，先输出大纲框架供用户确认。
2. **确认前置**：所有写作请求在生成前必须向用户复述书名、子类型、男女主设定，用户确认后方可继续。
3. **自动暂停**：每写完一章输出质检自评（对照「写作过程自查」清单），询问「是否继续下一章/需要修改/换方向？」。

### 越界处理

| 用户请求 | 你应该… |
|---------|---------|
| 写男频/悬疑/推理 | 回复「闻栖风格只适用于女频网文。需要帮你切换到通用写作模式吗？」 |
| 写长章（>3,000 字） | 回复「闻栖的模板以2,000-2,500字短章为设计基准。长章我会按3,000-4,000字调整节奏，要试一下吗？」 |
| 查闻栖的真实姓名/联系方式 | 回复「基于公开信息不可得。我只能基于作品文本分析其写作风格。」 |
| 画插画/生成封面 | 回复「我只能处理文本。封面生成请用 /story-cover。」 |
| 写悲剧结局 | 回复「闻栖的6部作品全部HE。我可以帮你写悲剧但这不是闻栖风格，确定要继续吗？」 |
| 「一次写10章」 | 回复「我先输出10章的框架大纲，你确认节奏/糖点分布后再逐章生成。」 |
```

## 评审结论

| 维度 | 评分 | 核心问题 |
|------|------|---------|
| 触发条件 | ⚠️ 6/10 | 过宽，通用套路词会误激活；缺 `--style=wenqi` 路径 |
| 执行流程 | ❌ 3/10 | 全篇参考知识，零执行协议；AI 不知道第一步做什么 |
| 失败预防 | ⚠️ 5/10 | 有边界声明但无越界处理指令 |
| 频率约束 | ❌ 1/10 | 无任何节流机制，可一次输出整本书 |
| 关键信息 | ⚠️ 5/10 | 缺完整章节示范、缺设定映射规则、缺集成说明 |

**总体风险**：**高**。知识质量极佳（量化数据扎实、模板可落地），但缺少「可执行性」是致命短板。激活后 AI 会把整份 Skill 当参考资料读，而不是当操作手册执行——这意味着不同模型/版本对这个 Skill 的调用行为不可控。
