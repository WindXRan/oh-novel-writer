## 最弱维度 1：工作流清晰度（得分 2/5）

### 问题描述

Skill 被激活后没有执行路径。现有结构是「参考手册」式组织（模型 → 启发式 → DNA → 模板 → 检查清单），AI 从顶部读到底部后不知道该先做哪步。工作量分散在 5 个模型 + 9 条启发式 + 4 套模板中，没有执行优先级和退出条件。

### 改进建议

#### 建议 1.1：在文首插入「执行入口」区块

**改前**：SKILL.md 第 7 行直接以 `# 闻栖 · 女频网文写作操作系统` 开头，「你是谁」之后没有任何执行指引。

**改后（插入在第 13 行「你是闻栖风格写作助手」之后）**：

```markdown
## 执行入口

接到用户请求后，按以下优先级判断行动路径：

路径 A — 用户提供了题材/角色设定
  → 直接进入「模板A：闻栖式黄金三章」填入内容
  → 完成后跑「黄金三章自查」
  → 用户确认后逐章展开（每章用「写作过程自查」）

路径 B — 用户说「帮我写个XXX」但无具体设定
  → 先用「模板B：闻栖式简介生成」产出简介+标签
  → 用户确认后再进入路径 A

路径 C — 用户要求修改/续写已写章节
  → 直接跳到「写作过程自查」逐项检验已有内容
  → 按每项检查输出修改清单，逐一执行
```

#### 建议 1.2：每条模型末尾添加「下一步行动」

**改前**：模型 4「三章一糖密度律」的结尾是「写一章时先检查——这章有糖点吗？上次糖点是几章前了？」——这是检查提示，不是行动指令。

**改后（追加在第 159 行之后）**：

```markdown
**下一步行动**：打开字数统计工具，翻看最近 3 章的糖点记录。
- 如果距离上次糖点 ≤ 3 章 → 保持，继续正常推进
- 如果距离上次糖点 = 4 章 → 本章必须安排一个小糖点
- 如果距离上次糖点 ≥ 5 章 → **立即停笔**，倒回上一章末尾插入一个糖点
```

#### 建议 1.3：将 4 个模板按执行优先级重排

**改前**：模板顺序为 A（黄金三章）→ B（简介）→ C（感情线节奏）→ D（番外）。但实际写作顺序通常是简介→开篇→展开→番外。

**改后**：将模板 B（简介生成）移到模板 A 之前，并在模板 C 末尾追加「中段乏力退出条件」：

```markdown
**中段退出判断**：写作过程中如果出现以下信号 → 暂停章节推进，先补事业线骨架：
- 连续 3 章写完后无人设推进
- 男女主场景重复（同一场景换台词写了 2 次以上）
- 糖点写完后没有新信息产生
```

## 最弱维度 2：失败预防（得分 1/5）

### 问题描述

全篇只有一条失败预防措施——模型 3 的「不知道怎么写一段时，优先写对话。对话不会错的。」其余地方只标注了「失效条件」（什么时候会出问题），但没有给出**恢复路径**（出问题后怎么办）。AI 在写作中遇到卡壳时没有可执行的兜底方案。

### 改进建议

#### 建议 2.1：新增「写作卡壳恢复手册」独立区块

**改后（插入在「质量检查清单」之后、「价值观与反模式」之前）**：

```markdown
## 写作卡壳恢复手册

### 场景 1：不知道怎么开头

```
恢复路径（从上到下，试到恢复为止）：
1. 写一个女主在困境中的画面（不用交代背景）
2. 写一句对话（谁在说话？说什么？不用解释为什么）
3. 写一个「没想到」的句子
4. 如果 3 步都不行 → 打开任意闻栖作品第 1 章，读前 300 字，然后关掉重写
```

### 场景 2：写到一半接不下去了

```
恢复路径：
1. 跳过当前段落，直接写本章结尾的信息钩（最后 3 句话）
2. 倒推：结确定尾巴后，从当前位置怎么走到这个结尾？
3. 用对话推进：让角色聊一件还没被聊过的事
4. 插入一个配角闯入场景打破僵局
```

### 场景 3：中段无力，发不出新内容

```
恢复路径：
1. 检查女主还有没有生存目标 → 如果没有，引入新外部威胁
2. 检查男主暗恋伏笔还剩多少 → 如果已用尽，插入一个他人视角的暗示
3. 字数阈值检测：距上次转折点超过 8 章？→ 安排一个小高潮
4. 最坏方案：让女配/前男友/婆婆找一次麻烦
```

### 场景 4：写完了但不满意

```
恢复路径（按顺序执行）：
1. 跑一遍「写作过程自查」6 项
2. 把最后一句话删掉，重写一个信息钩
3. 全文搜索「她/他」，找到最长的那段描写，改成对话
4. 如果还是不满意 → 另起一行写「别一版」，不从第一句改，重新写一遍
```

### 场景 5：用户反馈「不够闻栖」

```
恢复路径（按顺序执行）：
1. 检查「没想到」：最近 500 字里有吗？→ 没有就加一个
2. 检查对话占比：如果全部是叙述 → 挑一段改成对话
3. 检查女主内心 OS：「算了」「好吧」「懒得」出现了吗？
4. 检查结尾 3 句话：有信息钩吗？
5. 检查字数：2,200 字左右？
```

### 场景 6：虐完忘记发糖

```
恢复路径：
1. 立即停止当前章节
2. 在下一章开头 200 字内安排一个补偿性糖点
3. 糖点类型推荐：行动式（沉默地做了一件事）> 反差式 > 直接式
4. 补偿糖必须在虐后 2 章内，否则重新写虐段
```

#### 建议 2.2：失效条件后追加「恢复命令」而非仅标记风险

**改前**（第 90 行）：
```markdown
**失效条件**：短章只适合番茄等短章平台。如果目标平台是起点等长章平台，每章需要调整到 3,000-4,000 字。
```
→ 只说明了风险，没有给出行动。

**改后**：
```markdown
**失效条件**：短章只适合番茄等短章平台。如果目标平台是起点等长章平台，本模板需要调整。

**恢复命令**：
1. 将每章目标字数调整为 3,000-4,000 字
2. 每章内容从「单一事件」扩展为「事件 + 铺垫 + 收尾」三段结构
3. 糖点间距调整为每 5-7 章 1 个
4. 增加环境描写和配角支线占比（原模板的 1.5 倍）
```

#### 建议 2.3：质量检查清单追加「自动修复」项

**改前**：检查清单只有「是/否」判断，没有「如果否怎么办」。

**改后（修改第 400-406 行，追加修复指引）**：

```markdown
### 写作过程自查（每写一章）
- [ ] 这一章有糖点吗？（吻/抱/吃醋/心动细节）
      → 如果否：在第 40%-60% 位置插入一个男主不经意的关心动作
- [ ] 如果没糖点，离上次糖点超过 5 章了吗？
      → 如果是：在当前章节第 1 段就加一个糖点，不用铺垫
- [ ] 如果写了虐点，下 2 章内有补偿糖吗？
      → 如果否：在下一章标题就标明糖点，给读者预期
- [ ] 对话占了多少？如果全是叙述，改成对话试试
      → 替换方法：挑一段叙述 > 200 字的段落，改为「A: ... B: ...」形式
- [ ] 章节字数在 2,000-2,600 字之间吗？
      → 如果太长：找到对话最密集的段落拆成两章，断在悬念处
      → 如果太短：在结尾前加一段配角的对话或内心 OS
- [ ] 最后 3 句话有信息钩吗？
      → 如果否：删除最后一段的最后 2 句，重写为以下之一：
        突发信息 / 情绪反转 / 悬念抛出 / 外部入侵
```

## 总结

| 维度 | 当前评分 | 目标评分 | 关键改动 |
|------|---------|---------|---------|
| 工作流清晰度 | 2/5 | 4/5 | 插入执行入口 + 模型末尾加下一步行动 + 模板重排序 |
| 失败预防 | 1/5 | 4/5 | 新增卡壳恢复手册（6 场景）+ 失效条件后追加恢复命令 + 自查清单追加自动修复 |

两个弱维度的改进互不冲突，可并行实施。核心思路是将 SKILL.md 从「参考手册」重构为「可执行系统」：告诉 AI **先做什么、卡了做什么、做完检查什么、没通过怎么办**。
