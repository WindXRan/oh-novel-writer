# story-rewrite：结构仿写（全书模式）

**核心信念：只仿"骨相"，不仿"皮相"。**

---

## 流程总览

```
Phase 1（主线程）    读源文本 → 提取框架 + 设定锁 + 语感样本
Phase 2（全书写作）  创建项目 → 章纲预生成 → M agent并行×K章/agent → 循环至完本
Phase 3（收尾）      deslop + 禁用词扫描 + 一致性检查 + 字数校验
```

**默认直接写全书，不试水，不阻断。** 如需先试写3章预览，使用 `/story-rewrite-preview`。

---

## Phase 1：框架提取

主线程读源文本，提取：

1. **源文本分析**（不只是读前3章）：
   - 前3章：开篇方式、破冰方式、人设标签、叙述者风格
   - 中间转折点：读第1/4、1/2、3/4处各1章，提取节奏变化、情绪转折、冲突升级方式
   - 最后1章：结局走向、伏笔回收方式
   - 全书章数、卷结构、每卷核心事件
   - 提取3-5段语感样本（每段30-100字）+ 1章完整样本供agent深度模仿

2. **仿写框架**：新书名/主角/背景/核心冲突/开篇差异/破冰差异/节奏对齐表/角色名表
3. **设定锁**（所有agent共享的硬设定）：
   - 核心地名（具体地址）
   - 关键物件（来源+状态）
   - 人物关系锁（不可更改的关系）
   - 背景事件锁（统一的过去事件描述）
   - 时间线锁（各章时间约束）
4. **语感特征**：3-5个核心节奏特征（一句话描述+简短示例）

**框架落盘**：Phase 1 产出的框架保存到 `仿写试水库/框架_{书名}.md`（供 preview skill 复用）。

**反调色盘自检**：新书的"核心设定+核心冲突+角色原型"和源文本放一起对比，如果外人看了觉得"这不就是同一本书吗"，必须重写框架。具体维度：

| 维度 | 红线 |
|------|------|
| 核心设定 | 机制不能相同（只是换金额/地点不算差异化） |
| 核心冲突 | 驱动矛盾不能同类 |
| 角色原型 | 身份/处境/动机不能同类 |
| 开篇方式 | 第1章核心事件不能同类 |
| 破冰方式 | 第一次互动方式不能同类 |

## Phase 2：全书写作

Phase 1 完成后，**直接创建项目目录并开始写作**，不试水，不等用户确认。

### 断点续写（从 preview 继续）

如果 `仿写试水库/试水_{书名}.txt` 已存在（用户之前跑过 `/story-rewrite-preview`）：
1. 读取试水文件，提取已写的 3 章正文
2. 创建项目目录，将 3 章写入 `{书名}/正文/第001章_xxx.md` ~ `第003章_xxx.md`
3. 从第 4 章开始续写（跳过已写的 3 章）
4. 向用户汇报："已导入试水 3 章，从第 4 章开始续写"

### Step 1：创建项目 + 预生成章纲

```
1. 创建项目目录：
   {书名}/正文/
   {书名}/设定/（写入设定锁.md）
   {书名}/大纲/
   {书名}/追踪/（上下文.md / 伏笔.md / 时间线.md / 角色状态.md）

2. 章纲生成：
   - 根据源文本总章数确定新书章数（默认对齐，用户可指定缩放比例）
   - ≤30 章：spawn 1 个 story-architect，一次性出完
   - >30 章：spawn 2 个 story-architect，各出一半（前半本 / 后半本）
   - 主线程合并章纲，检查衔接：前半本末章的章尾钩子是否被后半本首章的章首钩子接住、关键人物/事件名是否一致
   - 写入 {书名}/大纲/章纲_全书.md
   - 若新书 >60 章，首轮只出前 50 章章纲，后续自动补充

3. 主线程检查章纲质量（钩子多样性、情节点具体性、情绪起伏）
4. 不通过的章 → 标记重写，通过的章进入写作队列
5. 并行度探测：spawn 5 个 agent → 成功则 +2 → 遇 429 则回退 → 锁定稳定值
6. **容错**：若一批中全部 agent 失败（429/超时/异常），降级为 K=1, M=3 重试一次；仍失败则暂停写作，报告用户
```

### Step 2：写作循环（每批 M 个 agent，每 agent 写 K 章）

**核心：每 agent 写 2-3 章（减少 spawn 开销 2-3 倍）**

```
Step 1：M 个 narrative-writer agent 并行 spawn
        每个 agent 负责连续 K 章（K=2 或 3，由主线程分配）
        prompt 注入：设定锁 + 语感样本 + K 章的章纲 + 上章摘要
        agent 用 write 工具依次写入 K 个文件：
          {书名}/正文/第{N}章_{章名}.md
          {书名}/正文/第{N+1}章_{章名}.md
          ...
        ⚠️ 不返回正文到主线程，直接写文件（省 token）

        并行度 M 自适应：
        - 默认 M=5, K=2 → 每批 10 章
        - API 好：M=5, K=3 → 每批 15 章
        - API 差：M=3, K=2 → 每批 6 章
        - 用户可配：--parallel=5 --per-agent=3
        - ⚠️ K 上限 = 3（超过 3 章会导致 agent 上下文溢出+质量下降）

Step 2：主线程批量字数校验（PowerShell 遍历本批文件）
        >3000 → 自动截断 | <1600 → 标记补写队列

Step 3（每 3 批执行一次）：
        - story-review 一致性审查（最近 2-3 批的章节）
        - 更新追踪文件（上下文.md / 伏笔.md / 时间线.md / 角色状态.md）

Step 3b（非 review 批次，轻量检查）：
        - grep 扫描本批文件的一级禁用词，发现即替换
        - grep 扫描设定锁关键地名/人名，发现不一致即修正

Step 4：检查剩余章纲，不足 2 批时 → story-architect 补充章纲

Step 5：字数不足的章节 → 加入下一批补写队列，和新章一起 spawn（不单独阻断）
```

**为什么每 agent 写多章**：
- 每次 spawn narrative-writer 要加载完整 system prompt（设定锁+语感样本+写作原则+参考文件路径）
- 10 章/agent×1 = 10 次 spawn = 10 次重复 prompt
- 5 agent×2 章/agent = 5 次 spawn = 省 50% prompt token
- 5 agent×3 章/agent = 5 次 spawn = 省 67% prompt token
- agent 内部写多章时，后续章节天然知道前章内容，不需要额外注入摘要

### 关键优化

**并行度自适应**
- 默认并行数：5（兼容大多数 API tier）
- 自动探测：首批 spawn 5 个 → 全部成功则 +2 → 直到出现 429/rate limit → 回退到上一个稳定值
- 推荐值：OpenAI Tier 1: 5 | Tier 2: 8 | Tier 3+: 10-15 | Anthropic: 5-10 | 第三方转发: 3-5
- 用户可配：`/story-rewrite --parallel=10` 手动指定

**审查降频**
- story-review 每 3 批跑 1 次（不是每批都跑）
- 只审查最近 2-3 批的章节（不是全书）
- 追踪文件每 3 批更新 1 次

**流水线执行**
```
Batch 1 写作 → Batch 1 字数校验+轻量检查 → Batch 2 写作 → Batch 2 字数校验+轻量检查 → ...
                                                        ↑
                                           同时：story-review(Batch 1) + 更新追踪
```
- 写作和审查错开执行，不互相阻塞
- 补写章节合并到下一批，不单独阻断
- 章纲预生成在第一批写作时就开始

**自动续写**
- Phase 1 完成后直接进入写作循环，不询问"是否继续"
- 用户可随时 Ctrl+C 中断，中断后已写内容完整保留

### 速度估算（200 章女频小说）

| 配置 | 每批产出 | 批次数 | 每批耗时 | 总耗时 |
|------|---------|--------|---------|--------|
| M=5, K=3（推荐） | 15 章 | 14 批 | 8-12 min | **2-3 小时** |
| M=5, K=2 | 10 章 | 20 批 | 6-10 min | **2.5-3.5 小时** |
| M=3, K=2（保守） | 6 章 | 34 批 | 5-8 min | **3-4.5 小时** |
| M=10, K=3（激进） | 30 章 | 7 批 | 12-18 min | **1.5-2 小时** |

## Phase 3：全书写完后

1. **全书 deslop**：对所有正文执行 story-deslop 的结构去AI（句长随机化、段落打散、句首分散、动作-反应解耦）
   - 200 章以上建议 spawn 3-5 个 deslop agent 并行处理（每 agent 负责 40-60 章）
   - 或按卷拆分，每卷一个 agent
2. **一致性终检**：角色属性/时间线/伏笔状态，发现矛盾修正（Step 3b 只做了轻量检查，这里是全书终检）
3. **字数总校验**：每章 >3000 自动截断

```
## 写作原则（按优先级执行）

你拥有以下参考文件，遇到具体问题时按需读取（路径相对于项目根目录）：
- `skills/story-long-write/references/style-craft.md`（镜头式写作、五感描写、文笔技巧、毒点规避）
- `skills/story-long-write/references/hooks-chapter.md`（章首7式+章尾13式+爽点设计）
- `skills/story-long-write/references/dialogue-mastery.md`（对话规则、权力博弈、人物语言差异化）
- `skills/story-long-write/references/emotional-arc-design.md`（六种弧线、期待感管理）
- `skills/story-long-write/references/commercial-core-methods.md`（卖点、节奏、情绪交付）
- `skills/story-long-write/references/anti-ai-writing.md`（去AI三遍法、7种模式检测、改写范例库）
- `skills/story-long-write/references/banned-words.md`（完整禁用词表）

> ⚠️ 以上路径相对于项目工作目录。用 read 工具读取，不要假设文件内容。

### ⚠️ 角色锚点（优先于所有写作原则，写错角色名=致命错误）
{角色名表：姓名/性别/身份/与主角关系}
{核心地名：具体地址}

### 1. 镜头式写作（最高优先级）
- 你是跟在主角身后的摄像机，只写能看到/听到/闻到/摸到的
- 不写"他心里想""他感到"，用动作和五感展示
- 每个段落=一个镜头，有明确的"拍摄对象"
- 五感：每个场景至少调动2种感官

### 2. 对话规则
- 每句对话必须推进剧情/塑造人物/制造张力之一
- 权力博弈：掌控者话短（≤10字），被动者话长（≥20字）
- 人物差异化：不同角色说话方式不同
- 口语化，不要书面腔

### 3. 情绪交付
- 不直接写"他愤怒/伤心/紧张"，用身体反应和行为展示
- 每章内部确定目的词（铺垫/高潮/爽点/打脸/人物塑造/设定），但不要写入正文
- 爽点公式：先不爽→制造希望→努力解决→得偿所愿
- 期待感：每章至少1个未闭环的期待

### 4. 章首钩子（选一种）
- 倒计时/意象预示/悬念对话/反差场景/闪前碎片/未完成动作/危机升级
- 同批内同类型钩子不超过2个

### 5. 章尾钩子（选一种）
- 突然揭示/紧急危机/未完成动作/两难抉择/神秘物品/意象钩子/留白
- 用动作或悬念收束，不要总结感悟

### 6. 去AI（必做）
- 禁用词：仿佛、犹如、宛若、如同、深吸一口气、缓缓、不禁、微微、轻轻、淡淡、眼中闪过、嘴角勾起、眉头微皱、心中一动、心头一震
- 句式：不用"脊背绷得笔直像XX""嘴角翘了一下不是笑是XX"等模板化比喻
- 段落：单句段≥30%，连续3段同句数必须打散（**对话段落豁免**，不计入句数统计）
- 句首：连续3句同主语开头必须换
- 用具体动作代替抽象比喻，用生活化比喻代替万能比喻
- **增量修改**：发现违规词/句时，只改那一句，不要全文重写

## 格式
第一行：`第{N}章 {章名}`
⚠️ 目的词是你的内部写作指导，不要写入正文。
用 write 工具直接写入文件：{书名}/正文/第{N}章_{章名}.md
```

## 字数自动修复

**统一用 PowerShell**（Windows 环境，避免 Python 编码问题）。

```powershell
# 批量校验本批章节（$batchFiles 由主线程从本批 spawn 的文件列表传入）
foreach ($file in $batchFiles) {
    $content = Get-Content $file -Raw -Encoding UTF8
    $chars = ($content -replace '[\s]','').Length
    if ($chars -gt 3000) {
        $paragraphs = $content -split '(\r?\n\r?\n|\r?\n)'
        $result = ""
        foreach ($p in $paragraphs) {
            $test = $result + $p
            if (($test -replace '[\s]','').Length -gt 2950) { break }
            $result = $test
        }
        $result | Set-Content $file.FullName -Encoding UTF8
        Write-Output "TRUNCATED: $($file.Name) -> 2950 chars"
    } elseif ($chars -lt 1600) {
        Write-Output "TOO_SHORT: $($file.Name) -> $chars chars (need补写)"
    } else {
        Write-Output "OK: $($file.Name) -> $chars chars"
    }
}
```

## 章纲质量标准

主线程出章纲后必须检查：
1. **钩子多样性**：同批内同类型章首钩子不超过2个
2. **情节点具体性**：必须写清"谁做了什么"，不能是"关系升温"这种空话
3. **情绪起伏**：每5章内至少1个情绪转折
4. 不通过 → 用 task_id 要求重写，告诉它哪些章不合格
