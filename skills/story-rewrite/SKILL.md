# story-rewrite：结构仿写（全书模式）

**核心信念：只仿"骨相"，不仿"皮相"。**

---

## 流程总览

```
Phase 1（主线程）    读源文本 → AI生成新书方案 → 提取框架 + 设定锁
Phase 2（全书写作）  创建项目 → 章纲预生成（断章自动补）→ M agent并行×K章/agent（缺文件自动补）→ 循环至完本
Phase 3（收尾）      deslop + 禁用词扫描 + 一致性检查 + 字数校验
```

**默认直接写全书，不试水，不阻断。** 如需先试写3章预览，使用 `/story-rewrite-preview`。

---

## Style Plugin 系统（可插拔风格）

`story-rewrite` 支持通过 `--style` 参数选配写作风格。风格由 **`story-style` skill** 统一管理，包含句式DNA、叙事规则、人物模板、质量检查表。

### 用法

```
/story-rewrite --style=wenqi       # 用闻栖风格写全书
/story-rewrite --style=wenqi --parallel=8 --per-agent=3
/story-rewrite                      # 不加--style用默认通用风格
```

### 风格加载路径

风格的注册、发现、injection 提取由 `story-style` skill 统一管理。`story-rewrite` 只负责调用：

**调用方式**：使用 `story-style` 的 injection 引擎脚本：
```bash
python skills/story-style/extract-injection.py --style={name}
```

**脚本输出**（JSON）：
- `style`: 风格名
- `label`: 中文标签
- `chapter_word_count`: `{min, target, max}` 字数配置
- `features`: 风格特征摘要
- `injections`: 各 key 的提取结果（`{target, content, found}`）

**加载流程**：
1. **调用脚本** → 获取 JSON 输出
2. **校验 `injections.voice.found`** → 为 false 则报错：`风格 '{name}' 的表达DNA缺失`
3. **注入到 prompt**：
   - `injections.voice.content` → 替换 prompt 中的 `{STYLE_VOICE}`
   - `injections.rules.content` → 追加到 prompt 中的 `写作原则` section
   - `injections.quality.content` → 替换 `质量检查` section（如有）
   - `injections.templates.content` → 注入到 `模板参考`（可选读取）
   - `injections.recovery.content` → 注入到 `写作卡壳恢复`（可选读取）
4. **`chapter_word_count`** → 替换 prompt 中的 `{STYLE_TARGET_WORDS}` 等变量

**错误处理**：
- 脚本返回 `error` 字段 → 报错并终止
- `injections.voice.found` 为 false → 报错：`风格 '{name}' 的表达DNA缺失`
- `injections` 全部为 false → 报错并终止

**⚠️ 必须指定风格**：不加 `--style` 时，询问用户选择风格：
```
请选择写作风格：
1. 闻栖（--style=wenqi）- 番茄女频·穿书/甜宠/反套路偏爱·吐槽型女主
2. 其他风格（请指定 --style=名称）

可用风格列表：/story-style
```

## Phase 1：框架提取

### 源文本要求

| 项目 | 要求 |
|------|------|
| 格式 | txt 或 md |
| 编码 | UTF-8 |
| 章节分隔 | 每章以 `第X章` 或 `第X章 章名` 开头（支持中文/阿拉伯数字） |
| 最低章数 | ≥ 3 章（少于 3 章无法提取有效骨架） |
| 来源 | 文件路径或粘贴文本均可 |

不满足要求时：提示用户修正格式后重试，不尝试自动修复。

主线程读源文本，提取：

1. **源文本分析**（不只是读前3章）：
   - 前3章：开篇方式、破冰方式、人设标签、叙述者风格
   - 中间转折点：读第1/4、1/2、3/4处各1章，提取节奏变化、情绪转折、冲突升级方式
   - 最后1章：结局走向、伏笔回收方式
   - 全书章数、卷结构、每卷核心事件
   - 提取3-5段语感样本（每段30-100字）+ 1章完整样本供agent深度模仿

2. **自动生成新书方案**（AI全自动）：
   读源文本后，根据源文本结构分析 + 风格插件（如有）自动生成新书信息，直接写入框架文件：
   - **书名**：结合源书爆款结构和风格特征，AI选最优
     - ⚠️ 有 `--style` 时：读取 `meta.injections.title`，按该风格的书名命名模式（结构模板+关键词池+情绪标签）生成书名
     - ⚠️ 无 `--style` 时：从源文本语感推断书名风格
   - **女主角名**：符合风格+题材的常用名，AI选最优
   - **男主角名**：AI选最优
   - **核心设定**：穿书/重生/先婚后爱等方向 + 背景风格，AI选最优
   - **章数**：对齐源书章数，默认不变
   - **题材**：风格插件自动锁定（现言/古言）

   ⚠️ 不询问用户，不弹选项，不阻断。AI自行决策后直接进入框架提取。

3. **设定锁**（所有agent共享的硬设定）：
   - 核心地名（具体地址）
   - 关键物件（来源+状态）
   - 人物关系锁（不可更改的关系）
   - 背景事件锁（统一的过去事件描述）
   - 时间线锁（各章时间约束）
4. **语感特征**：3-5个核心节奏特征（一句话描述+简短示例）

**框架落盘**：Phase 1 产出的框架保存到 `仿写试水库/框架_{书名}.md`（供 preview skill 复用）。

**反调色盘自检**：新书的"核心设定+核心冲突+角色原型"和源文本放一起对比，如果外人看了觉得"这不就是同一本书吗"，必须重写框架。具体维度：

| 维度 | 红线 |
|------|------|
| 核心设定 | 机制不能相同（只是换金额/地点不算差异化） |
| 核心冲突 | 驱动矛盾不能同类 |
| 角色原型 | 身份/处境/动机不能同类 |
| 开篇方式 | 第1章核心事件不能同类 |
| 破冰方式 | 第一次互动方式不能同类 |

## Phase 2：全书写作

Phase 1 完成后，**直接创建项目目录并开始写作**，不试水，不等用户确认。

### 断点续写（从 preview 继续）

如果 `仿写试水库/试水_{书名}.txt` 已存在（用户之前跑过 `/story-rewrite-preview`）：
1. 读取试水文件，提取已写的 3 章正文
2. 创建项目目录，将 3 章写入 `{书名}/正文/第001章_xxx.md` ~ `第003章_xxx.md`
3. 从第 4 章开始续写（跳过已写的 3 章）
4. 向用户汇报："已导入试水 3 章，从第 4 章开始续写"

### 断点续写（从中断处继续）

用户说"继续写"且 `{书名}/正文/` 已存在时：
1. 扫描 `{书名}/正文/` 目录，找到最大章节编号 N
2. 读取 `{书名}/大纲/章纲_全书.md`，确认总章数
3. 检查追踪文件完整性（上下文.md / 伏笔.md / 时间线.md / 角色状态.md）
   - 存在且非空 → 读取恢复上下文
   - 缺失或内容为空 → 从最近 3 章正文重建
4. 从第 N+1 章开始续写，跳过已完成的章节
5. 向用户汇报："已完成 {N} 章，从第 {N+1} 章继续"

### Step 1：创建项目 + 预生成章纲

```
⚠️⚠️⚠️ 核心规则：章纲必须由 story-architect subagent 生成，主线程禁止手写章纲 ⚠️⚠️⚠️

1. 创建项目目录：
   {书名}/正文/
   {书名}/设定/（写入设定锁.md）
   {书名}/大纲/
   {书名}/追踪/（上下文.md / 伏笔.md / 时间线.md / 角色状态.md）

2. 章纲生成（强制使用 subagent）：
   - ⚠️ **章数硬性对齐**：新书章数 = 源文本总章数，默认不变。例：源文191章 → 新书必须191章，不能多也不能少
   - ⚠️ **必须使用 story-architect subagent 生成章纲**，主线程不得手写章纲
   - ⚠️ **生成前必须记录源文本总章数**：在框架文件中明确写出"源文本总章数：X章，新书总章数：X章"
   - 章纲分批生成，每批最多50章：
     - ≤50 章 → spawn 1 个 story-architect subagent
     - 51-100 章 → spawn 2 个 story-architect subagent（前半/后半）
     - 101-150 章 → spawn 3 个 story-architect subagent
     - 151-200 章 → spawn 4 个 story-architect subagent
     - 以此类推，每50章一个subagent
   - ⚠️ story-architect 是 subagent 类型（`subagent_type: story-architect`），负责章纲生成。若 spawn 失败，主线程自行完成全部章纲生成
   - **章纲完整性校验**（核心新增）：各 agent 返回后，主线程检查每章是否有纲（第1章~第N章是否全覆盖）。发现章节断层（如 26-75 缺失）→ 自动 spawn 补写 agent，range = 缺失的起止编号，直到全部补齐
   - **合并章纲**：合并所有 agent 输出，检查衔接（章尾钩子接章首钩子、人物/事件名一致性）
   - 写入 {书名}/大纲/章纲_全书.md
   - ⚠️ **首轮必须生成全部章纲**，不能只出前50章。所有章纲生成完毕后才能开始写作

3. 主线程检查章纲质量（钩子多样性、情节点具体性、情绪起伏）
   - ⚠️ **第一步必须验证章数**：用grep统计章纲文件中的章数，与源文本总章数对比。不匹配 → 立即spawn补写agent
   - 发现空缺/不合格章 → 自动标记 + spawn 补写，不询问用户
4. 通过的章进入写作队列
5. 并行度探测：spawn 5 个 agent → 成功则 +2 → 遇 429 则回退 → 锁定稳定值
6. **容错**：若一批中全部 agent 失败（429/超时/异常），降级为 K=1, M=3 重试一次；仍失败则暂停写作，报告用户。暂停后用户说"继续写"时，从失败批次重新开始（断点续写逻辑）
```

**story-architect subagent 调用方式**：

⚠️ **强制流程**：章纲必须由 story-architect subagent 生成，主线程禁止手写章纲。

使用 Task tool spawn story-architect，示例：
```python
# 章纲生成调用（必须使用此方式，不能手写）
task(
    description="生成章纲 第1-50章",
    subagent_type="story-architect",
    prompt=f"""
你负责为仿写小说生成章纲（第1章 ~ 第50章）。

## ⚠️ 硬性要求
1. 必须生成完整的第1章到第50章章纲，不能遗漏任何一章

3. 核心事件必须写清"谁做了什么"，不能是"关系升温"这种空话
4. 章首钩子类型从以下选择：倒计时/意象预示/悬念对话/反差场景/闪前碎片/未完成动作/危机升级
5. 章尾钩子类型从以下选择：突然揭示/紧急危机/未完成动作/两难抉择/神秘物品/意象钩子/留白

## 设定锁
{设定锁内容}

## 源文本结构参考
{源文本的章纲/关键转折点}

## 语感特征
{3-5个核心节奏特征}

## 输出格式
严格按以下格式，每章一行，用 | 分隔：
第{{N}}章 | 章名 | 章首钩子类型 | 核心事件（谁做了什么）| 情绪基调 | 章尾钩子类型

示例：

## 质量要求
- 同批内同类型章首钩子不超过2个
- 每5章内至少1个情绪转折
- 章尾钩子必须制造未闭环的期待
```

### Step 2：写作循环（每批 M 个 agent，每 agent 写 K 章）

**核心：每 agent 写 2-3 章（减少 spawn 开销 2-3 倍）**

**narrative-writer 调用方式**（使用 Task tool）：
```python
# 单个 narrative-writer 调用
task(
    description="写第1-2章",
    subagent_type="narrative-writer",
    prompt=f"""
你负责写第1-2章。

## 设定锁
{设定锁内容}

## 语感样本
{语感样本}

## 章纲
### 第1章
{章纲1}

### 第2章
{章纲2}

## 上章摘要
{上章摘要（第1章无上章摘要）}

## 输出要求
- 直接用 write 工具写入文件，不返回正文
- 第1章写入：{书名}/正文/第1章_{章名}.md
- 第2章写入：{书名}/正文/第2章_{章名}.md
- 字数：每章 1600-3000 字
"""
)

# 多个 narrative-writer 并行调用
for batch in batches:
    for i, (start_ch, end_ch) in enumerate(batch):
        task(description=f"写第{start_ch}-{end_ch}章", subagent_type="narrative-writer", prompt="...")
```

```
Step 0：黄金三章特殊处理
        前3章必须由同一个 agent 写（保持开篇一致性）：
        - 第1批：1 个 agent 写第1-3章（K=3，不参与并行）
        - 第2批开始：正常并行（K=2 或 3）

Step 1：M 个 narrative-writer agent 并行 spawn
        每个 agent 负责连续 K 章（K=2 或 3，由主线程分配）
        prompt 注入：设定锁 + 语感样本 + K 章的章纲 + 上章摘要
        agent 用 write 工具依次写入 K 个文件：
          {书名}/正文/第{N}章_{章名}.md
          {书名}/正文/第{N+1}章_{章名}.md
          ...
        ⚠️ 不返回正文到主线程，直接写文件（省 token）

        **文件完整性校验**：所有 agent 返回后，主线程检查每章的.md是否实际存在。
        缺失的章 → 自动标记并 spawn 1 个补写 agent（K=缺失章数），不询问用户。
        连续缺失 >5 章 → 降 K=1 重试全批。

        并行度 M 自适应：
        - **推荐默认：M=5, K=3 → 每批 15 章**（速度最快，200章约2-3小时）
        - 质量优先：M=5, K=2 → 每批 10 章（质量更稳定，200章约2.5-3.5小时）
        - 保守模式：M=3, K=2 → 每批 6 章（API 差时使用）
        - 激进模式：M=10, K=3 → 每批 30 章（API 好时可尝试，200章约1.5-2小时）
        - 用户可配：--parallel=5 --per-agent=3
        - ⚠️ K 上限 = 3（超过 3 章会导致 agent 上下文溢出+质量下降）
        - ⚠️ M 上限 = 10（超过 10 会触发 API rate limit）

Step 2：主线程批量字数校验（PowerShell 遍历本批文件）
        >3000 → 自动截断 | <1600 → 标记补写队列

Step 3（每 3 批执行一次）：
        - story-review 一致性审查（最近 2-3 批的章节）
        - 更新追踪文件（上下文.md / 伏笔.md / 时间线.md / 角色状态.md）

Step 3b（非 review 批次，轻量检查）：
        - grep 扫描本批文件的一级禁用词，发现即替换
        - grep 扫描设定锁关键地名/人名，发现不一致即修正

Step 4：检查剩余章纲，不足 2 批时 → story-architect 补充章纲

Step 5：字数不足的章节 → 加入下一批补写队列，和新章一起 spawn（不单独阻断）
```

**为什么 K=3 是推荐配置**：
- **速度**：5 agent×3 章/agent = 5 次 spawn = 省 67% prompt token，200章约2-3小时
- **一致性**：同一 agent 写3章，后续章节天然知道前章内容，不需要额外注入摘要
- **容错**：K=3 时，单个 agent 失败影响3章，仍在可接受范围

**K=3 的风险与缓解**：
- 上下文溢出：3章+设定锁+语感样本+章纲可能超出 agent 上下文限制 → 缓解：精简 prompt，只注入必要信息
- 质量下降：agent 写到第3章时可能疲惫 → 缓解：设定锁+章纲提供清晰指引，减少 agent 自由发挥
- 只在 API 好且不追求极致质量时使用 K=3，否则降级为 K=2

### 关键优化

**并行度自适应**
- 默认并行数：5（兼容大多数 API tier）
- 自动探测：首批 spawn 5 个 → 全部成功则 +2 → 直到出现 429/rate limit → 回退到上一个稳定值
- 推荐值：OpenAI Tier 1: 5 | Tier 2: 8 | Tier 3+: 10-15 | Anthropic: 5-10 | 第三方转发: 3-5
- 用户可配：`/story-rewrite --parallel=10` 手动指定

**审查降频**
- story-review 每 3 批跑 1 次（不是每批都跑）
- 只审查最近 2-3 批的章节（不是全书）
- 追踪文件每 3 批更新 1 次

**流水线执行**
```
Batch 1 写作 → Batch 1 字数校验+轻量检查 → Batch 2 写作 → Batch 2 字数校验+轻量检查 → ...
                                                        ↑
                                           同时：story-review(Batch 1) + 更新追踪
```
- 写作和审查错开执行，不互相阻塞
- 补写章节合并到下一批，不单独阻断
- 章纲预生成在第一批写作时就开始

**自动续写**
- Phase 1 完成后直接进入写作循环，不询问"是否继续"
- 用户可随时 Ctrl+C 中断，中断后已写内容完整保留

**进度报告**
- 每批完成后输出进度摘要：`批次 {X}/{总批数} 完成，已写 {N}/{总章数} 章，当前并行度 M={M}, K={K}，预计剩余 {时间}`
- 429 错误时输出：`API 限流（429），已重试 {N} 次，并行度从 {M} 降级到 {M-2}，等待 {T} 秒后重试`
- 进度信息同步写入 `仿写试水库/进度_{书名}.txt`，方便用户监控

**质量衰减防护**
- 每 5 批（约 50 章）自动注入"风格变异指令"：强制要求换句式结构、换节奏模式，防止模式固化
- 变异指令不改变风格本质，只打散重复句式（如连续用"XX说"开头的对话 → 换成动作+对话混排）

### 速度估算（200 章女频小说）

| 配置 | 每批产出 | 批次数 | 每批耗时 | 总耗时 |
|------|---------|--------|---------|--------|
| M=5, K=3（推荐） | 15 章 | 14 批 | 8-12 min | **2-3 小时** |
| M=5, K=2 | 10 章 | 20 批 | 6-10 min | **2.5-3.5 小时** |
| M=3, K=2（保守） | 6 章 | 34 批 | 5-8 min | **3-4.5 小时** |
| M=10, K=3（激进） | 30 章 | 7 批 | 12-18 min | **1.5-2 小时** |

```
### 1. 镜头式写作（最高优先级）
- 你是跟在主角身后的摄像机，只写能看到/听到/闻到/摸到的
```

## 字数自动修复

**统一用 PowerShell**（Windows 环境，避免 Python 编码问题）。

```powershell
# 批量校验本批章节（$batchFiles 由主线程从本批 spawn 的文件列表传入）
foreach ($file in $batchFiles) {
    $content = Get-Content $file -Raw -Encoding UTF8
    $chars = ($content -replace '[\s]','').Length
    if ($chars -gt 3000) {
        $paragraphs = $content -split '(\r?\n\r?\n|\r?\n)'
        $result = ""
        foreach ($p in $paragraphs) {
            $test = $result + $p
            if (($test -replace '[\s]','').Length -gt 2950) { break }
            $result = $test
        }
        $result | Set-Content $file.FullName -Encoding UTF8
        Write-Output "TRUNCATED: $($file.Name) -> 2950 chars"
    } elseif ($chars -lt 1600) {
        Write-Output "TOO_SHORT: $($file.Name) -> $chars chars (need补写)"
    } else {
        Write-Output "OK: $($file.Name) -> $chars chars"
    }
}
```

## 章纲质量标准

主线程出章纲后必须检查：
1. **章数完整性（最重要）**：章纲必须覆盖第1章到第N章（N=源文本总章数），不能有遗漏。验证方法：用grep统计章纲文件中的"第.*章"行数，与源文本总章数对比。发现遗漏 → 立即spawn补写agent
2. **格式完整性**：每章必须包含5个字段（章名、章首钩子类型、核心事件、情绪基调、章尾钩子类型），缺一不可
3. **钩子多样性**：同批内同类型章首钩子不超过2个
4. **情节点具体性**：必须写清"谁做了什么"，不能是"关系升温"这种空话
5. **情绪起伏**：每5章内至少1个情绪转折
6. **动机合理性**：开局的核心偶遇/绑定事件（发帖被CEO看到、撞到霸道总裁等）必须自洽——检查"为什么是这个人/为什么是这样的方式"，避免"精准摇到顶级大佬"式悬浮。若不合理，加入"蓄谋已久"前提（男主不是偶然，是故意的），或在设定中给出合理解释
7. 不通过 → 用 task_id 要求重写，告诉它哪些章不合格。重写指令模板：
   ```
   第 {X}-{Y} 章章纲不合格，问题如下：
   - [ ] 章数缺失：缺少第 {A} 章、第 {B} 章……
   - [ ] 格式不完整：第 {N} 章缺少章尾钩子类型
   - [ ] 钩子重复：第 {N} 章与第 {M} 章同为「悬念对话」型
   - [ ] 情节点空泛：第 {K} 章"关系升温"→ 需改为具体事件（如"男主送女主回家，发现她住城中村"）
   - [ ] 动机不合理：第 {J} 章偶遇缺乏铺垫 → 加入"蓄谋已久"前提
   请重写这几章的章纲，保留章编号不变。
   ```
