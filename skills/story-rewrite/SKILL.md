# story-rewrite：结构仿写（全书模式）

**核心信念：只仿"骨相"，不仿"皮相"。**

---

## 流程总览

```
Phase 1（主线程）    读源文本 → AI生成新书方案 → 提取框架 + 设定锁
Phase 2（全书写作）  创建项目 → 章纲预生成（断章自动补）→ M agent并行×K章/agent（缺文件自动补）→ 循环至完本
Phase 3（收尾）      deslop + 禁用词扫描 + 一致性检查 + 字数校验
```

**默认直接写全书，不试水，不阻断。** 如需先试写3章预览，使用 `/story-rewrite-preview`。

---

## 上下文管理策略（解决大模型上下文不足问题）

### 核心原则

大模型上下文有限，长 prompt 容易丢失关键信息。解决方案：

1. **分层架构**：主线程负责调度和校验，子 agent 只负责单一任务
2. **状态管理**：关键信息存储在文件中，子 agent 通过 read 工具读取
3. **检查点机制**：在关键步骤设置检查点，发现问题立即修复
4. **摘要机制**：每个步骤结束后生成摘要，下一个步骤使用摘要

### 分层架构

```
主线程（调度+校验）
├── story-architect（章纲生成）
│   └── 只负责生成章纲，不负责写正文
├── narrative-writer（正文写作）
│   └── 只负责写正文，不负责章纲生成
├── story-review（一致性审查）
│   └── 只负责检查一致性，不负责修改
└── story-deslop（去AI味）
    └── 只负责去AI味，不负责其他
```

**主线程职责**：
- 读取源文本，提取框架
- 生成新书方案
- 创建项目目录
- 调度子 agent
- 校验子 agent 输出
- 管理状态文件

**子 agent 职责**：
- 只负责单一任务
- 通过 read 工具读取必要信息
- 不返回正文到主线程，直接写文件

### 状态管理

**关键信息存储在文件中**，而不是注入到 prompt 中：

| 文件 | 内容 | 读取方式 |
|------|------|---------|
| `{书名}/设定/设定锁.md` | 核心地名、关键物件、人物关系、背景事件 | 子 agent 通过 read 工具读取 |
| `{书名}/大纲/章纲_全书.md` | 全书章纲 | 子 agent 通过 read 工具读取 |
| `{书名}/追踪/上下文.md` | 当前写作上下文 | 子 agent 通过 read 工具读取 |
| `{书名}/追踪/伏笔.md` | 伏笔状态 | 子 agent 通过 read 工具读取 |
| `{书名}/追踪/时间线.md` | 时间线 | 子 agent 通过 read 工具读取 |
| `{书名}/追踪/角色状态.md` | 角色状态 | 子 agent 通过 read 工具读取 |

**prompt 中只注入必要信息**：
- 当前章节的章纲（不是全书章纲）
- 上章摘要（不是全书摘要）
- 角色名表（不是完整设定锁）
- 写作原则（不是全部参考文件）

### 检查点机制

在关键步骤设置检查点，发现问题立即修复：

| 检查点 | 检查内容 | 不通过时 |
|--------|---------|---------|
| 章纲生成后 | 章数是否完整、格式是否正确 | 自动 spawn 补写 agent |
| 正文写完后 | 字数是否达标、文件是否存在 | 自动标记补写队列 |
| 每3批后 | 一致性审查 | 自动修正矛盾 |
| 全书完成后 | deslop + 禁用词扫描 | 自动执行 |

### 摘要机制

每个步骤结束后生成摘要，下一个步骤使用摘要：

**章纲生成后**：
```
章纲摘要：
- 总章数：191章
- 卷结构：7卷（逃亡/暗涌/升温/危机/真相/追妻/结局）
- 关键转折点：第30章（苏婉凝出现）、第60章（傅司砚威胁）、第90章（记忆碎片）、第120章（真相揭露）、第150章（追妻开始）
- 爽点分布：每3-5章一个小爽点，每10-15章一个大爽点
```

**正文写完后**：
```
正文摘要：
- 已完成：50章
- 角色状态：温岁禾（紧张）、傅司寒（怀疑）
- 伏笔状态：U盘未找到、傅司砚未解决
- 时间线：第50章是来青峰村2个月
```

**下一个步骤使用摘要**：
- story-architect 使用章纲摘要生成后续章纲
- narrative-writer 使用正文摘要写后续正文
- story-review 使用摘要检查一致性

## Style Plugin 系统（可插拔风格）

`story-rewrite` 支持通过 `--style` 参数选配写作风格。风格由 **`story-style` skill** 统一管理，包含句式DNA、叙事规则、人物模板、质量检查表。

### 用法

```
/story-rewrite --style=wenqi       # 用闻栖风格写全书
/story-rewrite --style=wenqi --parallel=8 --per-agent=3
/story-rewrite                      # 不加--style用默认通用风格
```

### 风格加载路径

风格的注册、发现、injection 提取由 `story-style` skill 统一管理。`story-rewrite` 只负责调用：

**调用方式**：使用 `story-style` 的 injection 引擎脚本：
```bash
# 精简模式（推荐，适合注入prompt）
python skills/story-style/extract-injection.py --style={name} --compact

# 完整模式（适合调试）
python skills/story-style/extract-injection.py --style={name}
```

**脚本输出**（JSON）：
- `style`: 风格名
- `label`: 中文标签
- `chapter_word_count`: `{min, target, max}` 字数配置
- `features`: 风格特征摘要
- `injections`: 各 key 的提取结果（`{target, content, found}`）

**加载流程**：
1. **获取 style SKILL.md 路径** → 从 meta.json 的 source_skill 字段获取
2. **注入到 narrative-writer prompt** → 将 style_skill_path 注入到 prompt 中

**narrative-writer prompt 构造规则**：

1. **获取 style SKILL.md 路径**：
   ```python
   # 从 meta.json 获取 source_skill 路径
   import json
   with open("skills/story-style/wenqi/meta.json", "r", encoding="utf-8") as f:
       meta = json.load(f)
   style_skill_path = meta["source_skill"]  # 输出：skills/story-style/wenqi/SKILL.md
   ```

2. **注入到 prompt**：
   - `{style_skill_path}` → 替换为实际路径（如 `skills/story-style/wenqi/SKILL.md`）
   - narrative-writer 通过 read 工具读取此文件，获取完整风格信息
   - 口癖系统（有频率数据）
   - 章节开头/结尾模式

2. **构造 prompt**，按以下格式：
```
# ⚠️ 叙事视角：必须第三人称。

## 你的声音

### 句式特征
| 维度 | 数据 |
|------|------|
| 平均句长 | {数据} |
| 平均段长 | {数据} |
| 问号占比 | {数据} |
| 感叹号占比 | {数据} |

### 转折词系统（闻栖的叙事引擎）
几乎每段都含至少一个转折词：{转折词列表}

### 高频动词池
{动词列表，保留频率数据}

### 口癖系统（女主内心OS高频词）⚠️ 必须大量使用
{口癖列表，保留频率数据}

### 章节开头/结尾
{开头/结尾模式}

---

## ⚠️ 设定锁（不可违反）
请读取：{书名}/设定/设定锁.md

## 前情摘要
{前情摘要}

## 本次任务
你负责写 {K} 章（第{start}章 ~ 第{end}章），依次写入以下文件：
{文件列表}

每章独立成篇，但章与章之间要有连贯性：
- 第2章开头自然承接第1章结尾，不要重复已写内容
- 同一 agent 内后续章节不需要注入前章摘要（你能看到自己刚写的内容）
- 每章字数独立计算，各自满足 {min}-{max} 字

## 各章章纲
请读取：{书名}/大纲/章纲_全书.md → 找第{start}-{end}章章纲

## 写作原则（按优先级执行）
{从 style SKILL.md 提取的写作原则}

## 写作模板（强制应用）
{从 style SKILL.md 提取的写作模板}

## 格式
第一行：`第{{N}}章 {{章名}}`
⚠️ 目的词是你的内部写作指导，不要写入正文。
用 write 工具直接写入文件。

## 参考文件（按需读取）
{参考文件列表}

## Phase 1：框架提取

### 源文本要求

| 项目 | 要求 |
|------|------|
| 格式 | txt 或 md |
| 编码 | UTF-8 |
| 章节分隔 | 每章以 `第X章` 或 `第X章 章名` 开头（支持中文/阿拉伯数字） |
| 最低章数 | ≥ 3 章（少于 3 章无法提取有效骨架） |
| 来源 | 文件路径或粘贴文本均可 |

不满足要求时：提示用户修正格式后重试，不尝试自动修复。

主线程读源文本，提取：

1. **源文本分析**（不只是读前3章）：
   - 前3章：开篇方式、破冰方式、人设标签、叙述者风格
   - 中间转折点：读第1/4、1/2、3/4处各1章，提取节奏变化、情绪转折、冲突升级方式
   - 最后1章：结局走向、伏笔回收方式
   - 全书章数、卷结构、每卷核心事件
   - 提取3-5段语感样本（每段30-100字）+ 1章完整样本供agent深度模仿
   - **节奏特征提取**（必须）：
     - 爽点分布：统计每几章出现一个小爽点，每几章出现一个大爽点
     - 情绪弧线：总结前30章、31-60章、61-100章、101-150章、151-191章的基调
     - 开篇节奏：第1章必须有什么，第2章必须有什么，第3章必须有什么
     - 糖点密度：统计每几章一个糖点，虐完几章内必须发糖
     - 中段节奏：中段容易垮，需要用什么方式撑起来（事业线/配角线/冲突线）

2. **自动生成新书方案**（AI全自动）：
   读源文本后，根据源文本结构分析 + 风格插件（如有）自动生成新书信息，直接写入框架文件：
   - **书名**：结合源书爆款结构和风格特征，AI选最优
     - ⚠️ 有 `--style` 时：读取 `meta.injections.title`，按该风格的书名命名模式（结构模板+关键词池+情绪标签）生成书名
     - ⚠️ 无 `--style` 时：从源文本语感推断书名风格
   - **女主角名**：符合风格+题材的常用名，AI选最优
   - **男主角名**：AI选最优
   - **核心设定**：穿书/重生/先婚后爱等方向 + 背景风格，AI选最优
   - **章数**：对齐源书章数，默认不变
   - **题材**：风格插件自动锁定（现言/古言）

   ⚠️ 不询问用户，不弹选项，不阻断。AI自行决策后直接进入框架提取。

3. **设定锁**（所有agent共享的硬设定）：
   - 核心地名（具体地址）
   - 关键物件（来源+状态）
   - 人物关系锁（不可更改的关系）
   - 背景事件锁（统一的过去事件描述）
   - 时间线锁（各章时间约束）
4. **语感特征**：3-5个核心节奏特征（一句话描述+简短示例）

**框架落盘**：Phase 1 产出的框架保存到 `仿写试水库/框架_{书名}.md`（供 preview skill 复用）。

**反调色盘自检**：新书的"核心设定+核心冲突+角色原型"和源文本放一起对比，如果外人看了觉得"这不就是同一本书吗"，必须重写框架。具体维度：

| 维度 | 红线 |
|------|------|
| 核心设定 | 机制不能相同（只是换金额/地点不算差异化） |
| 核心冲突 | 驱动矛盾不能同类 |
| 角色原型 | 身份/处境/动机不能同类 |
| 开篇方式 | 第1章核心事件不能同类 |
| 破冰方式 | 第一次互动方式不能同类 |

## Phase 2：全书写作

Phase 1 完成后，**直接创建项目目录并开始写作**，不试水，不等用户确认。

### 断点续写（从 preview 继续）

如果 `仿写试水库/试水_{书名}.txt` 已存在（用户之前跑过 `/story-rewrite-preview`）：
1. 读取试水文件，提取已写的 3 章正文
2. 创建项目目录，将 3 章写入 `{书名}/正文/第001章_xxx.md` ~ `第003章_xxx.md`
3. 从第 4 章开始续写（跳过已写的 3 章）
4. 向用户汇报："已导入试水 3 章，从第 4 章开始续写"

### 断点续写（从中断处继续）

用户说"继续写"且 `{书名}/正文/` 已存在时：
1. 扫描 `{书名}/正文/` 目录，找到最大章节编号 N
2. 读取 `{书名}/大纲/章纲_全书.md`，确认总章数
3. 检查追踪文件完整性（上下文.md / 伏笔.md / 时间线.md / 角色状态.md）
   - 存在且非空 → 读取恢复上下文
   - 缺失或内容为空 → 从最近 3 章正文重建
4. 从第 N+1 章开始续写，跳过已完成的章节
5. 向用户汇报："已完成 {N} 章，从第 {N+1} 章继续"

### Step 1：创建项目 + 预生成章纲

```
⚠️⚠️⚠️ 核心规则：章纲必须由 story-architect subagent 生成，主线程禁止手写章纲 ⚠️⚠️⚠️

1. 创建项目目录：
   {书名}/正文/
   {书名}/设定/（写入设定锁.md）
   {书名}/大纲/
   {书名}/追踪/（上下文.md / 伏笔.md / 时间线.md / 角色状态.md）

2. 章纲生成（强制使用多 agent 并行）：
   - ⚠️ **章数硬性对齐**：新书章数 = 源文本总章数，默认不变。例：源文191章 → 新书必须191章，不能多也不能少
   - ⚠️ **必须使用 story-architect subagent 生成章纲**，主线程不得手写章纲
   - ⚠️ **生成前必须记录源文本总章数**：在框架文件中明确写出"源文本总章数：X章，新书总章数：X章"
   - ⚠️ **多 agent 并行生成**（必须）：章纲分批生成，每批最多50章，**必须并行 spawn**：
     - ≤50 章 → spawn 1 个 story-architect subagent
     - 51-100 章 → spawn 2 个 story-architect subagent（并行）
     - 101-150 章 → spawn 3 个 story-architect subagent（并行）
     - 151-200 章 → spawn 4 个 story-architect subagent（并行）

5. 通过的章进入写作队列
6. 并行度探测：spawn 5 个 agent → 成功则 +2 → 遇 429 则回退 → 锁定稳定值

**为什么 K=3 是推荐配置**：
- **速度**：5 agent×3 章/agent = 5 次 spawn = 省 67% prompt token，200章约2-3小时
- **一致性**：同一 agent 写3章，后续章节天然知道前章内容，不需要额外注入摘要
- **容错**：K=3 时，单个 agent 失败影响3章，仍在可接受范围

**K=3 的风险与缓解**：
- 上下文溢出：3章+设定锁+语感样本+章纲可能超出 agent 上下文限制 → 缓解：精简 prompt，只注入必要信息
- 质量下降：agent 写到第3章时可能疲惫 → 缓解：设定锁+章纲提供清晰指引，减少 agent 自由发挥
- 只在 API 好且不追求极致质量时使用 K=3，否则降级为 K=2

### 关键优化

**并行度自适应**
- 默认并行数：5（兼容大多数 API tier）
- 自动探测：首批 spawn 5 个 → 全部成功则 +2 → 直到出现 429/rate limit → 回退到上一个稳定值
- 推荐值：OpenAI Tier 1: 5 | Tier 2: 8 | Tier 3+: 10-15 | Anthropic: 5-10 | 第三方转发: 3-5
- 用户可配：`/story-rewrite --parallel=10` 手动指定

**审查频率**
- story-review 每 2 批跑 1 次（不是每批都跑）
- 只审查最近 2-3 批的章节（不是全书）
- 追踪文件每 2 批更新 1 次
- 字数异常章节每 2 批批量处理

**流水线执行**
```
Batch 1 写作 → Batch 1 字数校验+轻量检查 → Batch 2 写作 → Batch 2 字数校验+轻量检查 → Batch 3 写作 → ...
                                                        ↓
                                           story-review(Batch 1-2) + 更新追踪 + 字数异常处理
```
- 写作和审查错开执行，不互相阻塞
- 补写章节合并到下一批，不单独阻断
- 字数 <1600 的章节立即 spawn 补写 agent
- 章纲预生成在第一批写作时就开始

**自动续写**
- Phase 1 完成后直接进入写作循环，不询问"是否继续"
- 用户可随时 Ctrl+C 中断，中断后已写内容完整保留

**进度报告**
- 每批完成后输出进度摘要：`批次 {X}/{总批数} 完成，已写 {N}/{总章数} 章，当前并行度 M={M}, K={K}，预计剩余 {时间}`
- 429 错误时输出：`API 限流（429），已重试 {N} 次，并行度从 {M} 降级到 {M-2}，等待 {T} 秒后重试`
- 进度信息同步写入 `仿写试水库/进度_{书名}.txt`，方便用户监控

**质量衰减防护**
- 每 5 批（约 50 章）自动注入"风格变异指令"：强制要求换句式结构、换节奏模式，防止模式固化
- 变异指令不改变风格本质，只打散重复句式（如连续用"XX说"开头的对话 → 换成动作+对话混排）

### 速度估算（200 章女频小说）

| 配置 | 每批产出 | 批次数 | 每批耗时 | 总耗时 |
|------|---------|--------|---------|--------|
| M=5, K=3（推荐） | 15 章 | 14 批 | 8-12 min | **2-3 小时** |
| M=5, K=2 | 10 章 | 20 批 | 6-10 min | **2.5-3.5 小时** |
| M=3, K=2（保守） | 6 章 | 34 批 | 5-8 min | **3-4.5 小时** |

- **参考文件**：风格 SKILL.md、设定锁、章纲都存储在文件中
- **prompt 长度**：约 500-800 字（精简版）

## ⚠️ 写作前检查清单（每章写之前必须核对）
1. **读取风格 SKILL.md**：{style_skill_path} → 了解风格特征、写作原则、反模式

4. 读取上章正文：{书名}/正文/第{N-1}章_{章名}.md → 了解前情（第1章无需读取）

你负责写 {K} 章（第{N}章 ~ 第{N+K-1}章），依次写入以下文件：
{逐行列出：{书名}/正文/第{N}章_{章名}.md}

每章独立成篇，但章与章之间要有连贯性：
- 第2章开头自然承接第1章结尾，不要重复已写内容
- 同一 agent 内后续章节不需要注入前章摘要（你能看到自己刚写的内容）
- 每章字数独立计算，各自满足 1800-2600 字，目标 2200 字

## 参考文件（按需读取）

- `skills/story-long-write/references/banned-words.md`（禁用词表）
- `skills/story-long-write/references/anti-ai-writing.md`（去AI）
```

- `skills/story-long-write/references/dialogue-mastery.md`（对话规则、权力博弈、人物语言差异化）

> ⚠️ 若某个参考文件不存在，跳过该文件的指导，不报错。核心写作原则（镜头式写作、对话规则、情绪交付、去AI）已内嵌在本 prompt 中，参考文件只是补充。

### 1. 镜头式写作（最高优先级）
- 你是跟在主角身后的摄像机，只写能看到/听到/闻到/摸到的
- 不写"他心里想""他感到"，用动作和五感展示
- **例外**：女主内心OS/吐槽式旁白不受此限（属于人物声音而非叙述者声音，闻栖等女频风格尤需）
- 每个段落=一个镜头，有明确的"拍摄对象"
- 五感：每个场景至少调动2种感官

### 2. 对话规则
- 每句对话必须推进剧情/塑造人物/制造张力之一
- 权力博弈：掌控者话短（≤10字），被动者话长（≥20字）
- 人物差异化：不同角色说话方式不同
- 口语化，不要书面腔

### 3. 情绪交付
- 不直接写"他愤怒/伤心/紧张"，用身体反应和行为展示
- 每章内部确定目的词（铺垫/高潮/爽点/打脸/人物塑造/设定），但不要写入正文
- 爽点公式：先不爽→制造希望→努力解决→得偿所愿
- 期待感：每章至少1个未闭环的期待
- **阶级反差**：贫富/地位/阶级差距大时，用具体数字对比（工资 vs 合同金额、物价 vs 存款）+ 生理细节（手心出汗/脚顿住/心跳加速）写足震撼，不要轻描淡写一笔带过

### 4. 章首钩子（选一种）
- 倒计时/意象预示/悬念对话/反差场景/闪前碎片/未完成动作/危机升级
- 同批内同类型钩子不超过2个

### 5. 章尾钩子（选一种）
- 突然揭示/紧急危机/未完成动作/两难抉择/神秘物品/意象钩子/留白
- 用动作或悬念收束，不要总结感悟

### 6. 去AI（必做）
- 禁用词：仿佛、犹如、宛若、如同、深吸一口气、缓缓、不禁、微微、轻轻、淡淡、眼中闪过、嘴角勾起、眉头微皱、心中一动、心头一震
- 句式：不用"脊背绷得笔直像XX""嘴角翘了一下不是笑是XX"等模板化比喻
- 段落：单句段≥30%，连续3段同句数必须打散（**对话段落豁免**，不计入句数统计）
- 句首：连续3句同主语开头必须换
- 用具体动作代替抽象比喻，用生活化比喻代替万能比喻
- **排比句式**：连续"第一反应……第二反应……第三反应……"式排比≤1组，超出必须打散成长短句交替
- **增量修改**：发现违规词/句时，只改那一句，不要全文重写

⚠️ 目的词是你的内部写作指导，不要写入正文。
用 write 工具直接写入文件：{书名}/正文/第{N}章_{章名}.md
```

## 字数自动修复

**统一用 PowerShell**（Windows 环境，避免 Python 编码问题）。

**⚠️ PowerShell 中文编码问题**：
PowerShell 5.1 默认使用 GBK 编码，处理中文文件名/内容时会乱码。解决方案：

```powershell
# 方案1：设置 PowerShell 编码（推荐）
$OutputEncoding = [System.Text.Encoding]::UTF8
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8

# 方案2：使用 Python 替代（更可靠）
python -c "
import re
count = 0
with open('文件路径', 'r', encoding='utf-8') as f:
    for line in f:
        if re.match(r'^第\d+章', line):
            count += 1
print(f'章数: {count}')
"

# 方案3：使用 grep 工具替代 Select-String
grep -r "pattern" path/
```

```powershell
# 批量校验本批章节（$batchFiles 由主线程从本批 spawn 的文件列表传入）
foreach ($file in $batchFiles) {
    $content = Get-Content $file -Raw -Encoding UTF8
    $chars = ($content -replace '[\s]','').Length
    if ($chars -gt 3000) {
        $paragraphs = $content -split '(\r?\n\r?\n|\r?\n)'
        $result = ""
        foreach ($p in $paragraphs) {
            $test = $result + $p
            if (($test -replace '[\s]','').Length -gt 2950) { break }
            $result = $test
        }
        $result | Set-Content $file.FullName -Encoding UTF8
        Write-Output "TRUNCATED: $($file.Name) -> 2950 chars"
    } elseif ($chars -lt 1600) {
        Write-Output "TOO_SHORT: $($file.Name) -> $chars chars (need补写)"
    } else {
        Write-Output "OK: $($file.Name) -> $chars chars"
    }
}
```

## 章纲质量标准

主线程出章纲后必须检查：
1. **章数完整性（最重要）**：章纲必须覆盖第1章到第N章（N=源文本总章数），不能有遗漏。验证方法：用grep统计章纲文件中的"第.*章"行数，与源文本总章数对比。发现遗漏 → 立即spawn补写agent
2. **格式完整性**：每章必须包含5个字段（章名、章首钩子类型、核心事件、情绪基调、章尾钩子类型），缺一不可
3. **钩子多样性**：同批内同类型章首钩子不超过2个
4. **情节点具体性**：必须写清"谁做了什么"，不能是"关系升温"这种空话
5. **情绪起伏**：每5章内至少1个情绪转折
6. **动机合理性**：开局的核心偶遇/绑定事件（发帖被CEO看到、撞到霸道总裁等）必须自洽——检查"为什么是这个人/为什么是这样的方式"，避免"精准摇到顶级大佬"式悬浮。若不合理，加入"蓄谋已久"前提（男主不是偶然，是故意的），或在设定中给出合理解释
7. 不通过 → 用 task_id 要求重写，告诉它哪些章不合格。重写指令模板：
   ```
   第 {X}-{Y} 章章纲不合格，问题如下：
   - [ ] 章数缺失：缺少第 {A} 章、第 {B} 章……
   - [ ] 格式不完整：第 {N} 章缺少章尾钩子类型
   - [ ] 钩子重复：第 {N} 章与第 {M} 章同为「悬念对话」型
   - [ ] 情节点空泛：第 {K} 章"关系升温"→ 需改为具体事件（如"男主送女主回家，发现她住城中村"）
   - [ ] 动机不合理：第 {J} 章偶遇缺乏铺垫 → 加入"蓄谋已久"前提
   请重写这几章的章纲，保留章编号不变。
   ```
