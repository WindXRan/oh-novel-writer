# 甜宠文迭代报告：story-rewrite 反抄袭专项

## 迭代目标
让 story-rewrite SKILL.md 的输出不构成抄袭/洗稿，同时保持结构仿写的质量。

## 迭代轮次

| 轮次 | 问题 | 修复 | 效果 |
|------|------|------|------|
| Round 1 | 全部换皮抄袭（逐句搬运、场景1:1、角色映射） | Phase 2 禁止读源文本 | 逐句搬运归零 |
| Round 2 | T3/T4 仍结构性抄袭（框架太具体） | Phase 1 拆为 1a+1b，增加抽象化验证 | T3: S3→CLEAN, T4: S3→S2 |
| Round 3 | 验证修复效果 | — | T3 完全解决，T4 改进一个等级 |

## 关键发现

### 1. 三层防护缺一不可

| 防护层 | 作用 | Round 1 缺失 | Round 3 生效 |
|--------|------|-------------|-------------|
| Phase 1 抽象化 | 防止框架泄露故事梗概 | ❌ 框架=故事梗概 | ✅ 框架=叙事功能类型 |
| Phase 2 隔离源文本 | 防止 agent 照着写 | ❌ agent 能读源文本 | ✅ agent 无源文本路径 |
| Phase 3 抄袭检测 | 兜底检测 | ❌ 无检测环节 | ✅ 五层检测 |

### 2. Phase 1 抽象化是核心

Phase 2 禁止读源文本只解决了「逐句搬运」，但 agent 如果拿到的框架本身就是故事梗概（"重生→拯救前世为她死的人→慢慢接近"），它写出来的故事必然是源文本的换皮版本。

**真正的解决方案是 Phase 1 的两步拆分**：
- Step 1a：读源文本，输出原始分析（允许包含具体情节）
- Step 1b：只拿原始分析，做抽象化（不接触源文本），通过「抽象化测试」

### 3. 抽象化测试是关键验证

Step 1b 的核心机制是「抽象化测试」：

> 「如果把框架给一个完全没读过原文的人，他能据此写出3个完全不同但结构相似的故事吗？」

- 能 → 合格
- 不能 → 进一步抽象

Round 2 的框架："重生→拯救前世为她死的人"（只能写出1个故事 = 太具体）
Round 3 的框架："压抑/铺垫→转折→冲突→收束"（能写出无数个故事 = 合格）

## SKILL.md 最终架构

```
Phase 1（两步拆分）
├── Step 1a: agent 读源文本 → 输出原始分析
└── Step 1b: agent 只拿原始分析 → 抽象化验证 → 输出纯抽象框架

Phase 2（物理隔离）
├── agent 不接触源文本
├── 只接收抽象框架
└── 自行设计具体情节

Phase 3（抄袭检测）
├── 五层对比（结构/语句/人设/装置/句式）
└── REJECT 必须重写
```
