# 元迭代改进技能

让技能（SKILL.md）通过「生成 → 评估 → 分析 → 改进」的闭环循环自我进化。

## 触发条件

当用户说：
- `/meta-iterate`、`/迭代`、`/自我迭代`
- "迭代这个技能"、"让技能自己改进"
- "用迭代提升质量"、"持续改进"

## 工作流程

**核心行为**：迭代的是 SKILL.md 文件本身，不是生成的内容。

### 步骤 1: 确定迭代目标并收集输入

**自动完成**：
1. 读取用户提供的输入数据（源文本、范文等）
2. 分析输入数据的结构和特点
3. 设定迭代目标（基于技能描述自动推断）

**重要**：没有输入数据，无法进行真正的迭代！

### 步骤 2: 使用当前 SKILL.md 生成内容

- 使用当前版本的 SKILL.md 指导生成内容
- 保存生成结果到 `.meta-iteration/samples/`
- **关键**：生成的是内容，但迭代的是 SKILL.md

### 步骤 3: 评估生成的内容

评估维度（每项 0-100 分）：

| 维度 | 权重 | 评估要点 |
|------|------|----------|
| 内容质量 | 30% | 准确性、深度、相关性 |
| 结构完整性 | 25% | 组织性、连贯性、完整性 |
| 语言表达 | 25% | 流畅性、准确性、风格一致性 |
| 创新性 | 20% | 原创性、惊喜度、记忆点 |

### 步骤 4: 分析问题并更新 SKILL.md

**这是核心步骤！必须更新 SKILL.md 文件！**

1. 分析内容中的问题
2. 将问题转化为 SKILL.md 中的规则
3. 更新 SKILL.md 文件
4. 保存新版本到 `.meta-iteration/versions/v[N]/SKILL.md`

**示例**：
- 问题：内容开头平淡 → 更新 SKILL.md：添加"开头必须有钩子"规则
- 问题：对话缺乏金句 → 更新 SKILL.md：添加"每章至少2句金句"规则
- 问题：字数不均衡 → 更新 SKILL.md：添加"每章字数差异不超过20%"规则

### 步骤 5: 自动替换原技能文件

每次迭代后，**必须执行以下操作**：

```bash
# 1. 保存新版本存档
mkdir -p .meta-iteration/versions/v[N]
cp .claude/skills/[skill-name]/SKILL.md .meta-iteration/versions/v[N]/SKILL.md

# 2. 记录指标
echo '{"version":"v[N]","score":XX,"improvements":[...]}' > .meta-iteration/versions/v[N]/metrics.json

# 3. 【关键】自动替换原技能文件
# 将更新后的 SKILL.md 写入原技能目录，立即生效
cp .meta-iteration/versions/v[N]/SKILL.md .claude/skills/[skill-name]/SKILL.md
```

**重要**：步骤 3 是自动执行的，不需要用户确认。迭代完成后，原技能文件立即更新为最新版本。

### 步骤 6: 使用新 SKILL.md 重新生成内容

- 使用更新后的 SKILL.md 指导生成内容
- 保存生成结果到 `.meta-iteration/samples/`
- 评估新内容的质量
- 如果质量达标，迭代完成；否则返回步骤 4

### 步骤 7: 终止条件

迭代停止当：
- ✅ 质量分数 ≥ 阈值（默认 85）
- ✅ 连续 3 次改进幅度 < 2%（收敛）
- ✅ 达到最大迭代次数

### 步骤 8: 输出结果

```markdown
# 迭代完成报告

## 版本演进
| 版本 | 总分 | 关键改进 |
|------|------|----------|
| v001 | 72 | 基础版本 |
| v002 | 78 | 添加问题式开头 |
| v003 | 82 | 增加感官描写 |
| v004 | 85 | 优化节奏控制 ✅ |

## 最终版本
已更新 SKILL.md，保存在 .meta-iteration/versions/v004/

## 提取的成功模式
1. 问题式开头（置信度 92%）
2. 感官细节描写（置信度 88%）
3. 快慢节奏交替（置信度 85%）
```

## 仿写迭代发现的成功模式

通过实际使用 story-rewrite 技能迭代，发现以下成功模式：

### 1. 开头从系统出场开始（置信度 95%）
- **问题**：从日常对话开始，缺乏冲击力
- **解决**：直接从系统出场开始，更有悬念
- **示例**："手机屏幕黑了。不是没电，是突然黑了。"

### 2. 系统出场仪式感（置信度 90%）
- **问题**：系统出场平淡，缺乏视觉冲击
- **解决**：增加金色字体、屏幕黑屏、情绪反应
- **示例**："屏幕上浮出一行字。金色的字。"

### 3. 章尾钩子明确（置信度 85%）
- **问题**：章尾缺乏悬念，读者没有继续阅读的动力
- **解决**：以具体计划或冲突点结尾
- **示例**："第一步：买下整条商业街。"

### 4. 主角性格通过行为展现（置信度 80%）
- **问题**：直接描述性格，缺乏说服力
- **解决**：通过具体行为、对话、决策展现性格
- **示例**："他掐了自己一把，疼。不是做梦。"

## 上下文隔离规则（关键！）

### 问题描述
如果在同一个对话上下文中完成"分析问题 → 更新SKILL → 生成内容 → 评估"，
会把分析时获得的隐性知识（比如"这里要幽默"、"那里要删重复"）带入生成过程，
导致评分虚高——实际上SKILL.md本身并没有很好地指导这些改进。

### 解决方案：三阶段上下文隔离

#### 阶段A：分析与更新（主线程）
- 分析上一版本的问题
- 更新SKILL.md
- 保存新版本到 `.meta-iteration/versions/v[N]/SKILL.md`
- **结束后**：记录本轮要生成的内容要求（文件路径、字数等）

#### 阶段B：生成内容（独立子代理）
```
必须用 Task 工具启动一个"干净"的子代理来生成内容：
- prompt 中只传入 SKILL.md 的路径
- 不传入任何分析经验、问题描述、改进建议
- 子代理只读取 SKILL.md，按规则生成内容
- 生成完成后返回内容
```

#### 阶段C：评估内容（独立子代理）
```
必须用 Task 工具启动另一个"干净"的子代理来评估内容：
- prompt 中只传入生成的内容
- 不传入版本号、SKILL.md内容、之前的分析
- 子代理不知道这是第几版，不知道之前的分数
- 独立打分后返回评估结果
```

### 执行模板

```markdown
## 迭代 N 执行流程

### Step 1: 分析与更新（主线程）
[分析问题，更新SKILL.md]

### Step 2: 生成内容（子代理）
Task prompt:
"请根据以下SKILL.md生成内容：
SKILL路径：{path}
生成要求：{字数/章节等}
注意：只读取SKILL.md，不要参考任何其他信息。"

### Step 3: 评估内容（独立子代理）
Task prompt:
"请评估以下内容的质量，给出0-100的分数：
{生成的内容}
评估维度：内容质量、结构完整性、语言表达、创新性
注意：请独立打分，不要参考任何背景信息。"
```

### 验证方法
如果隔离成功：
- 相邻两个版本的分数差异应该 < 15分
- 如果差异 > 20分，说明存在上下文污染
- 分数应该呈现平稳上升趋势，而非跳跃式增长

## 注意事项

1. **备份原技能**：迭代前自动备份原始 SKILL.md
2. **渐进改进**：避免大幅修改导致退步
3. **保留核心**：不改变技能的核心定位
4. **可回滚**：保留所有版本，随时可回滚
5. **上下文隔离**：生成和评估必须用独立子代理，避免分数虚高
